# ActionHub-interview-qa-pack.md

# Espressif ActionHub｜三轮深挖问答包

> 使用方式  
> 不要逐字背答案，先记结构。  
> 推荐答题顺序：`先结论 -> 再原因 -> 再举例/数据 -> 最后复盘`

---

## 一、项目背景与业务价值

### Q1. 你为什么做这个项目？
因为内部项目型会议存在很明显的会后断点：纪要整理慢、行动项容易漏、Jira 还要再建一次。  
这个问题高频、可量化，而且和企业协同链路强相关。  
所以我选择从会后闭环切入，而不是做会中实时助手。

### Q2. 为什么你认为这是个真实需求，不是伪需求？
我先访谈了 `11` 位内部用户，并回放/旁听了 `7` 场会议。  
问题不是单点抱怨，而是在三类项目型会议里反复出现。  
我还做了两周基线采样，发现纪要整理中位时长是 `31` 分钟，这说明不是感受问题，而是真实效率问题。

### Q3. 为什么不做 To C 会议助手，而做内部项目协同？
因为内部项目协同更容易形成 `会议 -> 任务 -> 历史` 闭环。  
To C 会议助手往往停留在总结层，但内部项目协同能走到 Jira、Project History、Action Center。  
对 AI PM 来说，这类场景更能体现工作流和业务价值设计。

### Q4. 为什么只选三类会议？
0-1 产品最怕范围失控。  
这三类会议有共同结构：会后都有结论、行动项、风险或 blocker。  
同时它们又足够不同，能逼出模板和能力边界。

### Q5. 你怎么衡量这个项目是不是成功？
我分三层看：
- 效率：纪要整理时长、邮件发送时延
- 业务闭环：action item 采纳率、Jira 转单率
- AI 质量：owner/deadline 改写占比、badcase 回归通过率

我不会只看一个大而泛的“满意度”。

## 二、产品方案与关键取舍

### Q6. 为什么不是会中实时 AI？
因为最痛的点在会后，不在会中。  
会后更容易量化，也更适合先建立可信边界。  
会中实时对 ASR、时延、交互容错要求更高，不适合作为 MVP。

### Q7. 为什么不直接用 Teams 自带 recap？
Teams/Copilot 可以做 recap，但不能自然解决：
- 会议结果如何被结构化确认
- 哪些内容应该进入 Jira
- 哪些内容只进入邮件同步
- 历史会议如何按项目沉淀
- badcase 如何持续优化

所以 `ActionHub` 的价值不是 recap，而是把 recap 之后的协同链路补齐。

### Q8. 为什么只选一个主 transcript source？
MVP 更需要“一个可信版本”，而不是“最全版本”。  
多源融合会带来版本冲突、时间轴对齐、重复内容和归并错误。  
所以早期由主持人 / PM 先选一个主 source，更有利于后续 badcase 定位。

### Q9. 为什么 AI 只能起草，不能自动发邮件 / 建 Jira？
因为这是高影响写操作。  
企业场景里一旦误发、误建，信任成本会非常高。  
所以我选择“AI 生成草稿，用户确认后流转”，而不是全自动。

### Q10. 为什么 `owner / deadline` 要宁缺毋滥？
因为这两个字段一旦写错，用户信任损失远大于少填一个字段。  
我观察到用户对摘要错误的容忍度，比对 owner/deadline 错误高得多。  
所以没有证据就留空，并进入 `Needs Review`。

### Q11. 为什么先做 Jira draft，不直接自动建 Jira？
因为并不是所有 action item 都适合建 Jira。  
用户通常还需要确认 issue type、assignee、due date。  
如果直接建 Jira，会制造很多低质量任务。

### Q12. 为什么先做 Project History，再做 AI Ops？
因为历史沉淀先建立，后续跨会议能力才有基础。  
`AI Ops` 是平台运营视角，不是最早期用户最在意的价值。  
所以我先做业务价值，再做模型运营后台。

### Q13. 为什么 Phase 2 做 Skills，不做全自治 Agent？
因为 Skills 更容易定义边界、灰度、回滚和评测。  
自治 Agent 听上去更强，但在当前场景里会引入过高的业务和工程风险。  
我们需要的是稳定、可运营的能力包，而不是一个不可控的大脑。

### Q14. 为什么 RAG 放到 Phase 2？
MVP 先验证单次会议结构化是否有价值。  
如果过早引入 RAG，会同时增加历史数据质量、召回准确率、解释性和成本复杂度。  
我更倾向先把单次闭环跑通，再做跨会议召回。

### Q15. 为什么按会议类型拆模板，而不是一个总 prompt？
因为项目周会、技术评审会、风险同步会的结构差异很大。  
一个总 prompt 会让边界越来越模糊，后续很难稳定评测。  
拆模板以后，技术评审会把讨论误判成 decision 的问题明显下降。

## 三、AI / Workflow / 系统设计

### Q16. 如果让你 1 分钟讲清楚端到端运行链路，你会怎么讲？
我会先讲清输入是什么：会议对象、主 transcript、会议信息。  
然后讲 AI 在哪里介入：结构化抽取、摘要生成、路由建议。  
再讲边界：高风险写操作必须人工确认。  
最后讲回流：结果沉淀到历史和 action 视图，badcase 回到 AI Ops。

一句话版就是：  
`输入 -> AI 草稿 -> 人工确认 -> 下游流转 -> 数据沉淀 -> badcase 回流`

### Q17. 为什么你强调 `AI Draft / Needs Review / Confirmed / Confidence`？
因为 AI 企业产品的核心不是“有没有结果”，而是“用户敢不敢信”。  
用户最怕的是不知道哪些内容能用，哪些不能用。  
所以状态体系本质上是一个信任管理机制。

### Q18. 为什么不用一个大 prompt 直接输出所有内容？
因为不同会议类型结构差异很大。  
一个总 prompt 会让边界越来越模糊，后续无法稳定评测。  
所以我按会议类型拆模板、按能力模块拆任务，这是为了后续可优化、可回归。

### Q19. 你怎么平衡准确率、时延和成本？
我不会把所有问题都理解成“模型越强越好”。  
会后结构化是高价值链路，可以接受更高质量带来的稍高时延；  
但像路由建议、字段判断这类能力更适合规则或轻量判断，不一定每次都上最强模型。  
成本上优先复用已有链路，比如只做轻量历史召回，而不是完整 transcript 的重型向量检索。

### Q20. 为什么你说这不是单纯的模型项目，而是 AI 产品项目？
因为我定义的不只是模型输入输出，而是完整工作流。  
我同时设计了人工确认节点、状态体系、埋点、灰度门槛、回滚条件和 badcase 闭环。  
AI PM 的价值不只是让模型表现更好，而是让 AI 结果能被真实业务采用。

### Q21. 你们的 Knowledge Layer 是什么，不是知识库问答吧？
不是客服式知识库问答。  
它主要服务于：
- 模板路由
- 项目上下文补充
- 历史会议召回
- owner / 路由映射
- 跨会议 follow-up  
也就是说，它是协同系统的上下文层，而不是开放式 QA 系统。

### Q22. 你们为什么不做微调？
因为当前阶段数据量和 ROI 不支持一开始就微调。  
这个项目里的很多问题，本质上不是模型本体能力不足，而是：
- transcript 质量
- 模板设计
- 抽取边界
- 人工确认机制
- badcase 治理  
所以我优先做 Prompt、规则、模板和闭环。

## 四、结果、指标与 ownership

### Q23. 你在这个项目里到底负责什么？
我不是只写 PRD。  
我负责需求收敛、范围判断、核心流程设计、AI 能力边界、指标定义、原型推进和跨团队对齐。  
更重要的是：很多关键“不做什么”的决定是我拍板的，比如不做多 transcript 融合、不做自动发邮件、不做全自治 Agent。

### Q24. 你有没有做真实验证？
有，我做了两周基线采样和分阶段试点。  
总共覆盖 `107` 场会议，其中 MVP、Phase 1、Phase 2 分阶段推进。  
我会把不同阶段分开讲，不会把所有数据混在一起说成“全量上线结果”。

### Q25. 最好的结果是什么？
- 会后纪要整理中位时长从 `31` 分钟降到 `14` 分钟
- AI 草稿采用率达到 `72%`
- Action item 提取采纳率达到 `83%`
- Jira 草稿转正式任务率达到 `61%`
- 跨会议 follow-up 跟进率达到 `67%`

### Q26. 有没有没达标的？
有。  
`Project History` 早期周活只有 `38%`，没完全达到理想目标；  
blocker 识别命中率只有 `73%`，也没有完全打满。  
这些地方我不会回避，因为它们更能说明我做过真实复盘。

### Q27. 为什么 `Project History` 使用率不高？
因为它是沉淀型价值，不像纪要确认和发邮件那样是即时刚需。  
用户在高压项目里更关注“今天要处理什么”，不太会主动回看历史。  
这也是后来我把跨会议 follow-up 和 `Action Center` 提前的原因。

### Q28. 为什么 Jira 转正式任务率只有 61%，不算低吗？
单看这个数字会觉得一般，但它不能孤立理解。  
并不是所有 action item 都适合建 Jira。  
如果为了追高这个指标而把所有事项都 Jira 化，会制造大量无效任务。

### Q29. 你怎么证明不是模型太差，而是产品边界问题？
我们在复盘里发现，很多争议不来自摘要是否通顺，而来自 `decision`、`owner`、`deadline` 是否越权。  
所以我判断这不是单纯模型质量问题，更是产品风险控制问题。  
于是我通过状态体系、证据展示和人工确认去解决，而不是一味换模型。

### Q30. 这个项目和普通会议纪要工具的差异是什么？
纪要工具只解决“记录”。  
`ActionHub` 解决的是“记录 + 确认 + 发送 + 建单 + 追踪 + 优化”。  
它的核心不是一段总结，而是一条工作流。

### Q31. 你认为这个项目最体现你 AI PM 能力的点是什么？
我把一个 AI 能力问题，做成了一个可验证的业务闭环问题。  
这意味着我关注的不只是 prompt 和模型，而是：
- 用户痛点
- 产品边界
- 指标
- 风险
- badcase
- 演进路径

## 五、badcase、复盘与下一步

### Q32. 做过最关键的一个 trade-off 是什么？
我觉得是坚持“AI 先起草，人确认再流转”。  
因为这会牺牲部分自动化体验，但换来可控性和信任。  
在企业场景里，这是比“多自动一步”更重要的决策。

### Q33. 最失败的一次设计是什么？
早期技术评审会里把讨论误判成 decision，这是最明显的一次失败。  
它让我意识到不能用一个统一模板硬套不同会议。  
也是从那时开始，我推动按会议类型拆模板、按能力模块做评测。

### Q34. 你怎么定义 badcase？
不是所有编辑都是 badcase。  
我会区分：
- 正常润色
- 明确误提 / 漏提
- 幻觉补全
- 路由错误  
只有后几类才进入 badcase 池。

### Q35. badcase 是谁来收、谁来判、谁来推动修？
收集入口可以来自用户手动反馈、编辑记录、邮件/Jira 丢弃记录。  
初步分类通常由 PM 或 AI 运营角色来做。  
根因判断需要和算法、后端一起看，到底是数据问题、Prompt 问题、规则问题还是产品边界问题。  
最后由 PM 决定优先级和是否进入版本迭代。

### Q36. 你怎么判断一个 Skill 能不能上线？
我不会只看离线准确率。  
我会同时看：
- 离线集效果
- JSON 结构成功率
- badcase 回归
- 草稿采用率
- 字段改写率
- 失败率

只有这几项一起稳定，才会进入更大范围灰度。  
所以我把 AI 上线当成产品灰度，而不是单次模型实验。

### Q37. 如果再做一次，你会提前做什么？
- 更早定义 badcase taxonomy
- 更早结构化记录用户编辑行为
- 更早做轻量 Skill Registry
- 更早规划历史数据的召回边界和权限边界

### Q38. 下一步你会怎么演进？
- 继续优化跨会议 follow-up
- 增强 blocker pattern 识别
- 做更稳定的 Skill 版本治理
- 强化 Knowledge Layer 的历史上下文能力
- 在合适场景下再评估更深的自动化，而不是贸然提高写操作自动化率
